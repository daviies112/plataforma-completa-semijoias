# Nexus Core Operating Protocol 🛡️🚀

**Objective:** Maintain absolute consistency, security, and high performance across the entire Nexus ecosystem.

## 1. Environment & Shell Standards 🛠️
*   **Production First**: All updates, hotfixes, and corrections executed via the shell MUST use production mode.
    *   **Env Variables**: Always prefix commands with `$env:NODE_ENV="production"; $env:PORT="5000";` (on Windows/PowerShell) or equivalent to ensure the production environment is mirrored.
    *   **Verification**: After any build or restart, verify the server is running on the production port (default 5000).

## 2. Mandatory MCP Access 🧠
*   **Tool-First Approach**: Always check and utilize the available MCP servers (`n8n-automation`, `postgres-local`, `supabase-local`, etc.) before attempting manual or redundant operations.
    *   **Supabase Local**: Use for authentication, schema migrations, and user management tasks.
    *   **Postgres Local**: Use for core platform data and shared relational state.

## 3. Database Separation 🏗️
*   **Postgres Local**: This is the "Base for Everything." It handles the platform's primary relational data.
*   **Supabase Integrated Postgres**: Distinct from Postgres Local. Used specifically for Supabase's internal features (Auth, Realtime, Storage mappings).
*   **Credential Management**: Use `.env` and `supabase_config` table to manage cross-platform login connections.

## 4. Quality & Testing 🧪
*   **Exhaustive Testing**: Every task MUST conclude with an exhaustive testing phase.
    *   Use automated scripts (`node -e ...` or dedicated files) to verify database connectivity, API response tokens, and UI consistency.
    *   **No Placeholders**: Never use placeholder data in production fixes.

## 5. Proactive Communication 🗣️
*   **Explicit Clarification**: If a requirement is ambiguous or if a side effect is detected, STOP and ask the user.
*   **Proof of Work**: Document every significant change in `walkthrough.md`.
