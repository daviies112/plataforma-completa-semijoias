# Nexus Database Architecture & Management 📊🏗️

**Core Principle:** Strict separation between the application's base data and the authentication/infrastructure layer.

## 1. Database Landscape 🗺️
*   **Postgres Local (The Foundation)**:
    *   **Port**: 5432
    *   **Database**: `plataforma` (or as defined in `POSTGRES_LOCAL_URL`).
    *   **Role**: Primary storage for all business logic, tenant settings, and relational modules.
    *   **Management**: Handled via `db.ts` using Drizzle ORM and `postgres-local` MCP.

*   **Supabase Local (The Auth Layer)**:
    *   **Port**: 8100 (API) / 54322 (Integrated Postgres).
    *   **Database**: `postgres` (internal to Supabase).
    *   **Role**: User authentication, Realtime features, and PWA storage.
    *   **Management**: Handled via `@supabase/supabase-js` client and `supabase-local` MCP.

## 2. Integrated Postgres vs Local Postgres ⚠️
*   **Integrated Postgres**: This is the DB that comes *with* Supabase. It is NOT the same as your platform's Local Postgres.
*   **Avoid Collisions**: Do not store business logic tables in the Supabase Integrated Postgres unless they are strictly related to Supabase Features.

## 3. Login Synchronization 🔑
*   **The .env Strategy**:
    *   `SUPABASE_LOCAL_URL` and `SUPABASE_LOCAL_KEY` must be utilized by both the main platform and the store platform to synchronize login sessions.
    *   New users must have accounts provisioned in the Supabase Auth system and their metadata linked back to the `plataforma` Local Postgres.

## 4. Verification Procedures ✅
*   **Test Script**: Use `scripts/check_db_clients.ts` to verify that both the Postgres Local and Supabase instances are reachable and correctly populated.
*   **Connection Validation**: Before any query, ensure the correct client (Drizzle vs Supabase) is being used for the specific data domain.
